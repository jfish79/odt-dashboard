# On-Demand Transit Database

A searchable research inventory of microtransit and demand-response public transit deployments across the United States.

**[View the live dashboard →](https://jfish79.github.io/odt-dashboard/)**

> ⚠️ Replace `USERNAME` above (and in `index.html` OG tags) with your GitHub username after creating the repo.

---

## What's in here

| Path | Description |
|---|---|
| `index.html` | Interactive dashboard — Leaflet map, card view, analytics, reference tab |
| `ODT_Inventory.csv` | **Source of truth.** 241 systems across 24 states. Edit this to update the database. |
| `concerns.csv` | Data quality issues, vendor corrections, reclassification log |
| `legend.csv` | Column definitions and valid values |
| `summary.csv` | Auto-generated counts (regenerated by `build.py`) |
| `data/systems.json` | Generated from the CSV — consumed by the dashboard at runtime |
| `data/boundaries/` | Per-state GeoJSON service area overlays, loaded on demand |
| `data/boundaries/_index.json` | Auto-generated manifest of available boundary files |
| `scripts/build.py` | Build pipeline: CSV → `systems.json` + summary + boundary manifest |
| `scripts/migrate_classify.py` | One-time paratransit reclassification tool (provenance only) |
| `source/ODT_Database.xlsx` | Legacy spreadsheet snapshot. CSV is now the source of truth. |

## Workflow

### Updating data

1. Edit `ODT_Inventory.csv` (add rows, fix fields, update statuses).
2. Run the build: `python3 scripts/build.py`
3. Review the output summary, then commit both the CSV and regenerated files.

### Validating without writing

```
python3 scripts/build.py --check
```

Reports missing coordinates, unknown states, duplicate name+state pairs, and unusual system types.

### Adding boundary overlays for a new state

1. Place a GeoJSON file at `data/boundaries/{STATE}.geojson` (e.g. `TX.geojson`).
2. Each feature needs `properties.name` (matching a system name in the inventory), `properties.boundary_type`, and optionally `properties.source`, `properties.fidelity`, `properties.updated`.
3. Run `python3 scripts/build.py` — the manifest auto-updates.
4. The dashboard will load the overlay when that state is selected in the filter.

### Deploying to GitHub Pages

No build step required on the server. Push to `main`, then:

1. Go to **Settings → Pages → Source** → Deploy from branch: `main` / `root`.
2. The site is live at `https://USERNAME.github.io/odt-database/` within ~1 minute.

### Local testing

The dashboard fetches data at runtime, so it can't be opened via `file://`. Use a local server:

```
python3 -m http.server
```

Then open `http://localhost:8000`.

## Data quality and limitations

This is a research inventory, not an official government registry. Data is compiled from agency websites, press releases, state DOT studies, and the Federal Transit Administration.

**Known caveats (see `concerns.csv` for full details):**

- **120 supplemental CA/CO entries** were imported from external lists and have lower-confidence vendor data. The `Source` column in the CSV distinguishes original vs. supplemental rows.
- **83 paratransit services** (ADA/senior/disabled-restricted) are included but may be removed after review — they were reclassified from "Demand-Response" using a keyword classifier documented in `scripts/migrate_classify.py`.
- **Coordinates** for supplemental entries point to city centroids, not service zone centers.
- **Fleet, hours, and fares** change frequently. Verify with the operating agency before travel planning.
- **NC is the only state with service area boundary overlays.** Other states show pin markers only.
